**Descripción:**

## 📝 Notas y Aprendizajes

### Lecciones Técnicas

1. **Bcrypt es lento (intencionalmente)**: 12 rounds toma ~200ms, lo cual es BUENO para seguridad pero puede necesitar ajuste en ambientes con alto tráfico. Consideramos usar Argon2 en futuro.

2. **JWT size matters**: Incluir demasiados claims hace tokens grandes → headers HTTP grandes. Mantuvimos solo userId, email, roles.

3. **Refresh token rotation**: Inicialmente no rotábamos tokens. Tras investigar, implementamos rotación para prevenir replay attacks si un token es interceptado.

### Decisiones Arquitectónicas

**Por qué JWT en lugar de Sessions:**

*Alternativas consideradas:*
- Sessions server-side con Redis
- Sessions server-side con DB
- JWT stateless

*Decisión final:* JWT stateless

*Razones:*
- Escalabilidad horizontal sin estado compartido
- No requiere Redis adicional (simplifica infra)
- Fácil integración con mobile apps
- Estándar ampliamente adoptado

*Tradeoffs aceptados:*
- No se pueden revocar access tokens antes de expiración (mitigado con TTL corto de 24h)
- Tokens más grandes que session IDs
- Necesitamos refresh tokens en DB de todas formas (para revocación)

### Problemas Encontrados

**Problema 1: Tests fallando con "JsonWebTokenError: invalid token"**

- **Descripción:** Tests de middleware fallaban intermitentemente con error de token inválido
- **Causa:** Secreto JWT diferente en tests vs desarrollo (env variables mal configuradas)
- **Solución:** Creamos `.env.test` específico con secreto consistente para tests
- **Prevención:** Documentado en README: "Copiar .env.example a .env.test antes de ejecutar tests"
- **Commit:** `t6u7v8w` - fix(test): configure JWT secret for test environment

**Problema 2: Refresh tokens no se invalidaban al hacer refresh**

- **Descripción:** Los refresh tokens antiguos seguían funcionando después de hacer refresh
- **Causa:** Olvidamos implementar la lógica de invalidación del token anterior
- **Solución:** Añadimos campo `used: true` al refresh token anterior en DB al generar uno nuevo
- **Prevención:** Añadimos test específico: "should invalidate old refresh token after rotation"
- **Lección:** Tests de seguridad son críticos, no opcionales
- **Commit:** `x9y0z1a` - fix(auth): invalidate old refresh token on rotation

**Problema 3: Bcrypt rounds excesivos causaban timeout en tests**

- **Descripción:** Tests de registro tardaban >30s, timeout de Jest
- **Causa:** Usábamos 12 rounds en tests (mismo que producción)
- **Solución:** Variable de entorno `BCRYPT_ROUNDS` con valor 4 para tests, 12 para prod
- **Prevención:** Documentado en guía de configuración
- **Commit:** `b2c3d4e` - perf(test): use lower bcrypt rounds in test environment
